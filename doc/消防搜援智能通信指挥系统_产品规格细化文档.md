## 一、产品定位与核心价值主张

### 1.1 一句话定位

> **以现有自组网通信设备为核心，通过 AI 图像识别、语音事件化、多源感知融合与任务闭环能力，将消防搜援现场的"感知碎片"转化为"可指挥的结构化信息流"。**

### 1.2 解决的本质问题

| 现有痛点 | 本系统解决方式 |
|---------|--------------|
| 视频回来了，但看不过来 | AI 自动识别重点目标，主动推送告警 |
| 发现了人，但上报靠喊话 | 语音事件结构化，自动生成事件卡 |
| 不知道哪里搜过、哪里没搜 | 搜救热力图 + 已搜/未搜网格 |
| 队员状态不透明、异常发现晚 | 腕表持续采集 + 异常自动置顶告警 |
| 主链路断了什么都不知道 | 低轨窄带保底，关键状态不断链 |
| 任务下发后无法追踪 | 任务闭环 + 腕屏回执 + 超时提醒 |

### 1.3 核心差异化七点（对外宣传口径）

1. **AI 识别驱动指挥**：视频不是终点，识别结果才是行动入口
2. **现有底座直接升级**：不推翻设备，叠加行业价值增强
3. **多源融合感知**：视频 + 热成像 + 毫米波 + 语音 + GIS 一体化
4. **三端统一接入**：笔记本 + PAD/手机 + 腕屏，角色分明
5. **两类角色分明**：一线少操作、强提示；指挥员看全局、抓重点
6. **队员安全明确可见**：腕表 + 状态监测 + 失联保护 + SOS 协同
7. **断网不断报**：Mesh 主链路 + 低轨窄带双保险

## 二、目标用户画像

### 2.1 角色矩阵

| 角色 | 英文代号 | 作业终端 | 核心诉求 | 操作约束 |
|-----|---------|---------|---------|---------|
| 一线消防员 | **FF** | 腕屏 + 开放耳机 + 腕表 | 接令快、上报快、让我知道下一步做什么 | 戴手套、视线受阻、高压 |
| 班组长 | **TL** | 手持台 + PAD + 热成像 + 腕屏 | 掌握小队全局、热成像辅助判断、快速派任务 | 轻度手持可用 |
| 现场指挥员 | **IC** | PAD / 笔记本 | 全局态势、AI 筛选、任务闭环追踪 | 需要一个界面看清全局 |
| 总指挥 / 后方 | **CC** | 笔记本 | 战略决策、多部门协同、汇报留痕 | 远程，对细节容忍度低 |
| 通信保障人员 | **CT** | 笔记本 链路管理页 | 保障链路稳定、异常快速处置 | 技术型用户 |
| 采购/管理层（演示受众）| **BU** | — | 看到亮点、理解价值、支持采购决策 | 非技术，需直觉冲击 |

### 2.2 一线消防员（FF）详细画像

- **作业环境**：室内浓烟、高温、噪音 >85dB、视线 <1m
- **装备负荷**：全套防护 >20kg，手部操作极受限
- **信息接收偏好**：语音播报优先 > 腕屏震动提示 > 简短视觉提示
- **主要恐惧**：迷路断联、队友失联无人知晓、任务不明确
- **期望系统行为**：系统主动推送而非需要主动查询

### 2.3 班组长（TL）详细画像

- **作业环境**：与前者相似，但承担小队协调责任
- **信息需求**：需要同时关注 3-6 名队员状态
- **核心操作**：分配任务、确认告警、热成像判断、语音指挥
- **期望系统行为**：小队状态一屏可见；告警自动聚合不刷屏

## 三、系统整体架构

### 3.1 五层架构（细化）

```
┌─────────────────────────────────────────────────────────────┐
│  Layer 5: 中心智能层（视觉大模型 + 事件分析 + 复盘引擎）         │
├─────────────────────────────────────────────────────────────┤
│  Layer 4: 指挥平台层（笔记本端 + PAD端 / 融合指挥系统升级）       │
├─────────────────────────────────────────────────────────────┤
│  Layer 3: 边缘智能层（手持台/背负台/车载节点上的本地推理能力）     │
├─────────────────────────────────────────────────────────────┤
│  Layer 2: 现场网络层（Mesh 自组网主链路 + 低轨窄带保底副链路）     │
├─────────────────────────────────────────────────────────────┤
│  Layer 1: 前端采集层（视频/热成像/毫米波/语音/腕表/腕屏/GPS）     │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 数据类型与传输优先级

| 数据类型 | 主链路优先级 | 窄带保底 | 说明 |
|---------|-----------|---------|------|
| SOS告警 | P0 最高 | ✅ 必须保障 | 生命安全 |
| 队员位置 | P0 最高 | ✅ 必须保障 | 心率+坐标每30s |
| 重要结构化事件 | P1 | ✅ 支持 | AI识别结果/语音事件卡 |
| 任务指令 | P1 | ✅ 支持 | 下发/回执 |
| 视频关键帧快照 | P2 | 有条件 | 仅AI识别命中时推送 |
| 实时视频流 | P3 | ❌ 不支持 | 主链路专属 |
| 语音转写全文 | P3 | ❌ 不支持 | 主链路专属 |

**交互细节：**
- GIS 地图：支持楼层切换（B1/1F/2F/3F），队员位置实时刷新（≤3s）

- 告警列表：P0 告警自动置顶+闪烁+声音，点击跳转对应视频/地图
- 人员状态卡：异常队员（心率>150或静止>3分钟）自动置顶红框高亮

```
│                                         │  └──────────────────┘  │
├─────────────────────────────────────────┴────────────────────────┤
│  右侧备选视频宫格（4路）：点击切换到主窗口                          │
```

**目标卡片状态机：**
- `待确认` → 指挥员点「确认」→ `已确认`
- `已确认` → 点「派任务」→ 弹出任务编辑框 → `已派发`
- `已派发` → 执行队员回执 → `处理中` → `已完成`

#### 4.1.4 任务闭环管理页

**任务生命周期：**
```
[创建来源] → (AI识别自动生成 / 指挥员手动创建 / 语音上报触发)
     ↓
[任务草稿] → 补充：任务类型/执行人/目标位置/优先级/截止时间
     ↓
[已下发] → 消息推送到对应队员腕屏/PAD
     ↓
[已接收] → 队员回执确认（腕屏一键操作）
     ↓
[执行中] → 可语音上报进展
     ↓
[待验证] → 语音上报"任务完成" → 自动进入待验证
     ↓
[已完成] / [超时告警] / [取消]
```

**超时规则：**
- 下发后 3 分钟无回执 → 黄色提醒班组长
- 下发后 5 分钟无回执 → 红色告警指挥员
- 执行超时（可自定义）→ 自动告警并建议补派

#### 4.2.2 PAD 端设计原则

- **最小点击目标**：≥ 60px × 60px（戴手套可触达）
- **层级不超过3层**：从首页到任何功能最多3次点击
- **语音优先**：屏幕任何位置双击可激活语音上报
- **颜色语义统一**：
  - 🔴 红色 = 紧急告警/SOS/异常
  - 🟡 橙色 = 警告/待处理/超时
  - 🟢 绿色 = 正常/完成/已确认
  - 🔵 蓝色 = 信息/已派发/进行中
- **离线容忍**：断网时仍可查看最后缓存任务和地图，能发SOS

### 4.3 腕屏端

#### 4.3.1 信息分级显示规则

**腕屏尺寸约束**：屏幕面积 ≤ 手机的 1/4，只显示最关键三层信息：

| 层级 | 触发条件 | 显示内容 | 持续时间 |
|-----|---------|---------|---------|
| **L1 常态** | 无告警 | 我的任务区域编号 + 剩余时间 + 网络状态 | 持续 |
| **L2 提醒** | 新任务/语音播报/位置提示 | 简短文字 + 震动1次 | 10s后返回L1 |
| **L3 告警** | 队友SOS/指挥告警/失联 | 全屏红色 + 文字 + 持续震动 | 手动确认 |

#### 4.3.2 腕屏核心交互（物理按键优先）

```
腕屏操作方式（按键优先，减少触屏需求）：
- 短按侧键1：语音上报启动
- 长按侧键1（2秒）：SOS触发
- 短按侧键2：回执当前任务
- 双击屏幕：查看队伍状态
- 上滑：下一条信息
- 下滑：上一条信息
```

#### 4.3.3 腕屏视频查看规格

- **帧率**：最大 5fps（低轨窄带下降为图片轮播模式）
- **触发方式**：指挥员主动推送给指定队员 或 队员手动请求
- **内容类型**：重点视频快照 / 热成像截图 / AI识别结果截图
- **展示时长**：自动 30s 后返回常态界面

## 五、典型场景完整用户旅程

### 5.1 场景一：城市建筑火灾内攻搜救

**背景**：某居民楼3层发生火灾，2层有被困人员报告，现场浓烟，一个班组（1名TL + 4名FF）已进入内攻。

#### Phase 1：进场前（T-5min）

| 步骤 | 角色 | 行为 | 系统响应 |
|-----|-----|-----|---------|
| 1 | CT | 开启现场 Mesh 自组网，确认节点覆盖 | 系统显示组网拓扑图，节点在线 |
| 2 | IC | 笔记本端创建任务：「3号楼内攻搜救」，标注建筑平面图 | GIS加载楼层平面图，网格初始化为「未搜」 |
| 3 | IC | 将任务下发给班组长TL | TL的PAD收到推送，腕屏震动提醒 |
| 4 | TL | 确认任务，将区域拆分下发给FF-01~04 | 各队员腕屏显示对应区域编号 |
| 5 | FF | 腕屏收到任务：「负责2F-A区」，一键回执接收 | 任务状态变为「已接收」 |

#### Phase 2：内攻搜救中（T+0 ~ T+20min）

| 步骤 | 角色 | 行为 | 系统响应 |
|-----|-----|-----|---------|
| 6 | FF-01 | 进入2层，胸挂视频自动开始回传 | 边缘节点收流，AI开始实时识别 |
| 7 | 系统 | AI识别到「疑似倒地人员」，置信度78% | 指挥台右侧弹出目标卡；告警声响；CC看板标记 |
| 8 | IC | 点击目标卡「确认」，在地图上标记位置 | 疑似目标在GIS地图落点，状态「待处理」 |
| 9 | IC | 点击「派任务」，选择距离最近的FF-02 | FF-02腕屏震动，显示：「前往2F-东北方向，疑似被困者」 |
| 10 | FF-02 | 侧键2回执接收任务 | 任务变为「执行中」 |
| 11 | FF-01 | 语音上报：「发现一名男性，意识清醒，请求担架」 | ASR识别→自动生成事件卡（类型:发现被困者/状态:清醒/需求:担架/位置:2F+GPS） |
| 12 | IC | 事件卡出现在时间线，系统建议：「联系后方担架组」 | IC确认后一键生成资源请求任务 |

### 6.2 语音智能系统规格

#### 语音事件结构化提取字段

当语音满足触发条件（关键词或主动上报模式），系统提取以下字段：

| 字段 | 示例值 | 提取方式 |
|-----|-------|---------|
| 事件类型 | 发现被困者 / 发现火点 / 请求支援 | 意图识别 |
| 数量 | 1名 / 3名 | NER数量词 |
| 位置描述 | 2层东北角 / 楼梯旁 | NER位置词 |
| 目标状态 | 意识清醒 / 无意识 / 受伤 | 意图+状态词 |
| 资源请求 | 担架 / 医疗队 / 破拆设备 | 资源词典 |
| 时间戳 | 自动记录 | 系统时间 |
| 来源设备 | FF-01-手持台 | 设备ID |

**触发模式：**
- **主动模式**：用户说「上报」/「报告」后，进入5秒结构化识别窗口
- **被动监听**：持续监听呼救词列表（「救命」「帮帮我」「有人吗」等）
- **指令模式**：识别操作指令（「截图」「派任务给TL-01」「标记此位置」）

#### 语音系统性能要求

| 指标 | 要求 |
|-----|-----|
| ASR 识别准确率 | 嘈杂环境下 ≥ 85% |
| 端到端延迟（本地推理）| ≤ 800ms |
| 呼救词检测召回率 | ≥ 90%（宁可多报，不可漏报）|
| 支持方言 | 普通话为主，支持接入方言模型扩展 |
| 离线可用性 | 本地 ASR 模型运行于设备 AP 端，断网可用 |

#### ⚠️ 重要硬件架构说明：ASR 不运行在 Mesh 板卡上

> **Mesh 板卡（自组网通信模块）的职责是射频通信与网络协议处理，其 CPU/RAM 不足以运行 ASR 模型，强行运行会干扰通信稳定性。**

现场设备普遍采用**双芯片/双板架构**：

```
┌──────────────────────────────────────────────────────┐
│  手持台 / 背负台 物理设备                              │
│                                                      │
│  ┌───────────────────┐    ┌────────────────────────┐ │
│  │  应用处理器（AP）   │    │     Mesh 通信板卡       │ │
│  │  Android / Linux  │    │  (RTOS / 专用固件)      │ │
│  │                   │    │                        │ │
│  │  ✅ 音频采集        │    │  ✅ 射频发送/接收       │ │
│  │  ✅ 前端降噪        │    │  ✅ Mesh 自组网协议     │ │
│  │  ✅ 本地 ASR 推理   │←──→│  ✅ 传输文字(bytes级)  │ │
│  │  ✅ 关键词触发检测   │    │  ✅ 数据路由/加解密     │ │
│  │  ✅ 离线缓存结果    │    │  ❌ 不处理音频/ASR      │ │
│  └───────────────────┘    └────────────────────────┘ │
└──────────────────────────────────────────────────────┘
```

**带宽收益**：AP 先做 ASR 转为文字，再经 Mesh 板卡传输文字结果，而非传输原始音频流：
- 原始音频：约 32~64 KB/s（8kHz 8bit AMR编码）
- ASR 文字结果：约 30~200 字节/句
- **带宽节省约 99%**，大幅降低 Mesh 网络压力

#### 各设备 ASR 能力分级

| 设备类型 | AP 计算能力 | ASR 部署方式 | 上限模型 |
|---------|-----------|-----------|---------|
| **手持台** | 中等（4核 ARM A55 级）| 本地轻量 ASR | Whisper-tiny 级 |
| **背负台** | 较强（可扩展 AI 模块）| 本地标准 ASR + 呼救词检测 | Whisper-base 级 |
| **车载台 / 指挥台** | 强（PC 级 x86/A76）| 完整 ASR，含方言扩展 | Whisper-small 及以上 |
| **PAD / 手机** | 强（骁龙 7/8 系）| 完整线上/线下 ASR | Whisper-small 级 |
| **腕屏** | 极弱（MCU 级）| ❌ 不运行 ASR | 仅接收文字推送 |
| **腕表** | 极弱 | ❌ 不运行 ASR | 仅接收震动/简短文字 |
| **Mesh 板卡** | 弱（RF 专用）| ❌ **禁止放置 ASR** | 会破坏通信稳定性 |

#### 弱 AP 设备的 ASR 代理机制

对于 AP 能力不足（或无 AP）的设备（如仅有 Mesh 板卡的中继节点），语音通过以下方式处理：

```
音频采集（设备麦克风）
    ↓
通过 Mesh 网络传输压缩音频（低优先级，有带宽才发）
    ↓
边缘 ASR 代理节点处理（背负台 or 车载台 承担推理）
    ↓
结果文字回传给来源设备（极小数据量）
    ↓
来源设备显示转写结果
```

**代理节点选择策略**：
- 优先选择**信号最强的背负台**作为区域内 ASR 代理节点
- 车载台作为整个任务区域的**ASR 兜底节点**
- 代理关系在任务启动时自动协商，节点失联时自动换代理

### 6.2-A 语音转写（ASR Transcription）完整规格

#### 功能定位

语音转写是语音子系统的**底层能力**，将现场所有语音频道的音频实时转为可检索、可存档的文字流。它不做语义理解（语义理解是语音事件结构化的工作），只做准确转录。

#### 转写数据流

```
麦克风/耳机采集音频
    ↓
前端降噪处理（本地，< 50ms）
    ↓
本地 ASR 模型推理（边缘设备，< 800ms）
    ↓
转写文字带时间戳上传至平台
    ↓
平台层分流：
  ├── 语音频道实时字幕显示（指挥台底栏）
  ├── 关键词触发检测（呼救词 / 指令词 / 上报触发词）
  └── 全文存档（可搜索语音日志）
```

```
└─────────────────────────────────────────────────────┘
```
- 单行显示最新一条，超出省略号
- 点击「▶展开」→ 上滑弹出半屏转写历史
- 关键词橙色高亮

**腕屏端**：不显示转写文字（屏幕太小），仅接收基于转写结果生成的指令/任务推送。

#### 转写全文检索规格

指挥台可对本次任务所有转写记录进行搜索：
- 支持关键词搜索（人名/地点/资源词）
- 支持时间段筛选
- 点击搜索结果 → 定位到对应时间的视频片段（联动语音与视频）
- 支持导出：TXT / SRT字幕格式 / JSON结构化

### 6.2-B 语音指令（Voice Command）完整规格

#### 设计原则

> 语音指令不是语音助手。它只识别**固定意图的短句**，不做自由问答，确保在高噪环境下指令识别率最高、误触率最低。

#### 指令激活方式

| 终端 | 激活方式 | 说明 |
|-----|---------|-----|
| 手持台 / 耳机 | 说「指令」或「命令」+ 等待提示音 | 避免误触 |
| PAD（班组长）| 长按语音按钮 1 秒 + 等待提示音 | |
| 腕屏 | 短按侧键1 + 等待震动 | 物理键激活 |
| PC端（指挥台）| 键盘快捷键 `Alt+V` + 等待提示音 | 辅助操作 |

激活后系统进入「指令监听窗口」（5 秒），超时自动关闭。

#### 指令集完整定义

**一、上报类指令（FF / TL 使用）**

| 指令话术 | 触发动作 | 示例 |
|---------|---------|------|
| `「上报」/ 「报告」` | 进入结构化上报窗口（5秒录音 + 解析）| 「上报」→ 说出内容 |
| `「发现被困者」` | 直接生成「发现被困者」类事件卡（需补充细节）| 快捷触发 |
| `「发现火点」` | 生成「发现火点」类事件卡 | — |
| `「请求支援」` | 生成资源请求任务（定位自动填入）| — |
| `「请求担架」` | 生成「请求担架」资源任务（自动指定后方担架组）| — |
| `「请求破拆」` | 生成「请求破拆设备」任务 | — |
| `「任务完成」` | 当前任务标记为「待验证」→ 提示IC确认 | — |

**二、位置类指令（FF / TL 使用）**

| 指令话术 | 触发动作 |
|---------|---------|
| `「标记位置」` | 将当前GPS坐标 + 当前视频截图生成「位置标记」卡，同步到GIS |
| `「标记火点」` | 在GIS地图落点「疑似火点」，橙色图标 |
| `「标记危险区」` | 在GIS当前位置附近生成「危险区」多边形（需说出范围或后台估算） |
| `「我在 [位置描述]」` | 更新自身位置文字标注（与GPS联合定位） |

**三、查询类指令（IC / TL 使用）**

| 指令话术 | 触发动作 |
|---------|---------|
| `「FF-03状态」` | 语音播报：FF-03当前心率/位置/任务状态 |
| `「未搜区域」` | 语音播报本次覆盖率 + 未搜区域数量 |
| `「当前告警」` | 语音播报当前未处理P0/P1告警数量 |
| `「链路状态」` | 语音播报当前主链路/窄带状态 |

**四、操作类指令（IC 使用，PC端/PAD端）**

| 指令话术 | 触发动作 |
|---------|---------|
| `「截图」` | 对主视频窗口截图，保存到任务日志 |
| `「派任务给 [角色ID]」` | 打开任务预填单，执行人自动设为指定角色 |
| `「切换 [FF/TL 编号] 视频」` | 将该队员的视频切到主窗口 |
| `「全员集合」` | 向所有腕屏推送「全员集合」震动提示 + 文字 |
| `「紧急撤离」` | 触发全员撤离指令，所有终端红色告警 + 震动 |

#### 指令识别状态机

```
[待机]
  ↓ 激活词触发
[指令监听] ← 5秒超时 → [超时关闭，返回待机]
  ↓ 识别到有效指令
[确认播报] ← 指令不确定（置信度<75%） → [播报「请重复」+ 重新监听]
  ↓ 置信度≥75%
[执行指令] → 语音反馈「已执行：XXX」
  ↓
[返回待机]
```

#### 指令执行反馈规范

所有指令执行后，系统必须有**语音反馈**（不能只有视觉反馈，因为用户可能看不到屏幕）：

| 场景 | 语音反馈示例 |
|-----|-----------|
| 指令执行成功 | 「好的，已生成发现被困者事件卡」 |
| 指令识别失败 | 「没有听清，请重复指令」 |
| 指令无权限 | 「抱歉，该指令需要班组长权限」 |
| 指令执行超时 | 「系统繁忙，请稍后重试」 |
| 任务标记完成 | 「任务已上报完成，等待指挥员确认」 |

**语音反馈规格**：
- TTS 语速：适中，紧急反馈语速略快
- 音量：随环境噪声自适应（需设备麦克风反馈）
- 腕屏无扬声器时：仅震动 + 屏幕文字提示代替语音反馈

### 6.2-C 语音上报（Voice Reporting）完整交互流程

#### 上报类型分类

| 类型 | 触发方式 | 结构化程度 | 适用场景 |
|-----|---------|----------|---------|
| **结构化上报** | 说「上报」后按模板描述 | 高度结构化 | 有时间组织语言时 |
| **快捷上报** | 说固定短语（「发现被困者」）| 中度结构化 | 紧急，信息简短 |
| **自由语音上报** | 随时说任何内容（系统提取关键词）| 低度结构化 | 极端紧张，无法标准化 |
| **被动呼救检测** | 无需主动触发，系统监听 | 仅关键词 | 失去行动能力时 |

#### 结构化上报完整交互（主流场景）

```
Step 1：FF-03 激活上报
  - 说「上报」或长按手持台PTT键
  - 系统：提示音 ✓ + 腕屏震动1次 + 屏幕显示「🎙 正在录音...」

Step 2：FF-03 描述上报内容（5秒录音窗口）
  - 例：「发现一名老年女性，意识不清，腿部受伤，在2层东北角卫生间旁」

Step 3：本地 ASR 转写（< 800ms）
  - 转写结果显示在FF-03的手持台屏幕上
  - 腕屏同步显示（确认状态）

Step 4：本地 NLP 解析（< 500ms）
  - 提取：{ 类型:发现被困者, 数量:1, 描述:老年女性, 状态:意识不清+腿伤, 位置:2F东北卫生间 }

Step 5：系统确认播报
  - 耳机播报：「已识别：发现1名被困者，老年女性，意识不清，2层东北，确认上报请说是」
  - 腕屏显示：简短摘要 + [✅确认] / [✏重新说]

Step 6：FF-03 确认
  - 说「是」/ 按腕屏确认键
  → 事件卡上传至平台（< 1秒）
  → IC指挥台右侧弹出「新事件卡：发现被困者」
  → IC耳机提示音 + 屏幕提示

Step 7（可选）：IC处理
  - IC可以补充信息、确认、派任务
  - 任务派出后 FF-03 腕屏震动 + 显示「已派出支援」
```

#### 快捷上报交互（极简场景）

```
FF-03 说：「发现被困者」

系统（< 2秒）：
  - 自动从设备簇获取位置信息
  - 生成基础事件卡：{ 类型:发现被困者, 位置:当前GPS+楼层, 时间:自动, 来源:FF-03 }
  - 上传平台 + 通知IC
  - 耳机反馈：「好的，已上报发现被困者，位置已自动记录」
  - IC告警：「FF-03 发现被困者，待补充详情」

补充细节（IC侧）：
  - IC可通过语音频道追问FF-03补充细节
  - 补充内容自动追加到已有事件卡
```

#### 自由语音上报（兜底方案）

当FF无法按标准格式上报时（极端紧张/受伤/嘈杂），系统进入自由转写模式：

```
FF-03 说：「这里…走廊里…有个人…躺着不动…二楼…」（断断续续）

系统处理流程：
  1. 全文转写并存档
  2. 关键词提取：{ 地点:走廊/二楼, 状态:躺着不动 }
  3. 生成「低置信度待确认」事件卡（附原始语音链接）
  4. IC收到提示：「FF-03 语音上报（低置信度）- 需人工确认」
  5. IC可点击事件卡回放原始语音 + 查看转写文字
  6. IC手动补全信息后确认
```

**低置信度事件卡样式**：
- 左边框颜色：黄色（区别于正常橙色/红色）
- 标题后附 `[低置信度·需确认]` 橙黄色徽章
- 卡片内含：[▶ 播放原始语音] 按钮

#### 被动呼救词检测（无需主动触发）

系统持续（静默）监听所有语音频道，检测以下词汇：

**呼救词词库（可配置）**：
```
高优先级（单词即告警）：
  救命、SOS、快来、不行了、出不去、被压住了、断气

中优先级（需2词共现）：
  有人 + 帮帮我
  救援 + 快 / 帮助 + 快
  不行 + 人 / 这里 + 帮

环境音辅助（配合AI识别）：
  疑似哭声 + 敲击声 → 提升周围区域搜索优先级
```

**触发后处理**：
```
检测到呼救词（置信度 ≥ 60%）
  ↓
系统自动：
  1. 在语音转写栏高亮该句（橙色背景）
  2. 生成「疑似呼救」告警，推送给IC + TL
  3. 在GIS地图标记声源估算位置（来源设备GPS + 传播方向估算）
  4. 不自动生成事件卡（需指挥员确认，避免误报）

指挥员操作：
  [确认是呼救] → 转为「发现生命迹象」事件卡 + 派任务
  [忽略]       → 标记为「已核实-非呼救」，保留记录
```

#### 多人同时说话处理规范

| 场景 | 处理方式 |
|-----|---------|
| 同频道多人同时说话 | 按信号强度优先转写首选信号，其他标记「混音-部分转写」|
| 紧急指令（IC紧急广播）| 自动插队，打断当前转写队列，即时执行 |
| TL和FF同时上报 | 分优先级：TL先处理；FF内容入队列，标记「处理中」|
| 呼救词与正常对话同时 | 呼救词检测独立通道，不受正常对话影响 |

### 6.2-D 语音频道管理规格

#### 频道架构

```
全局频道架构：
  ├── 指挥频道（IC ↔ 所有TL）    ← IC讲话，所有TL接收
  ├── 班组频道（TL ↔ 本组FF）    ← TL讲话，本组FF接收
  ├── 全员广播频道（单向）        ← IC/CC讲话，所有设备接收（紧急使用）
  └── 跨频道呼叫（点对点）        ← 任意角色呼叫任意个人
```

#### 频道切换规则

| 角色 | 默认收听频道 | 说话发送频道 | 可主动切换到 |
|-----|-----------|-----------|-----------|
| FF | 班组频道 | 班组频道 | 仅班组频道 |
| TL | 班组频道 + 指挥频道 | 两频道可选 | 全员广播（需权限）|
| IC | 指挥频道 + 全员监听 | 指挥频道 | 全员广播，点对点 |
| CC | 全员监听 | 全员广播，指挥频道 | 所有频道 |

**转写标注频道来源**：每条转写记录标注来源频道颜色，让IC一眼区分哪个频道的信息。

#### PTT（按键发话）vs 开放监听模式

| 模式 | 适用场景 | 切换方式 |
|-----|---------|---------|
| PTT模式（按键发话）| 常规作业，减少杂音 | 默认模式，按住说话 |
| HOT MIC（开放监听）| 腕屏端（手太忙）、紧急时 | 设置中开启；或被IC强制开启 |
| 静默模式 | 潜伏/危化侦察 | TL/IC安排，仅接收不发送 |

**IC强制开启队员HOT MIC**（紧急查岗场景）：
- IC发现FF-03超时无响应 → 点击「强制查听」
- FF-03耳机提示「指挥员已静默查听」（让队员知道）
- 持续 30 秒后自动关闭，回到PTT模式

### 6.3 队员安全保障体系规格

#### 生命体征告警规则

| 指标 | 正常范围 | 一级警告 | 二级告警 | 三级紧急 |
|-----|---------|---------|---------|---------|
| 心率 | 60~150bpm | >150持续1min | >170持续30s | >190或<40 |
| 血氧 | ≥95% | 92%~94% | 88%~91% | <88% |
| 静止时长 | — | >2分钟无运动 | >4分钟无运动 | >6分钟无运动 |
| 体表温度 | 36~38°C | >38.5°C | >39.5°C | >40.5°C |

**告警分发逻辑：**
- 一级警告 → 推送给TL（安静提示）
- 二级告警 → 推送给TL + IC（明显提示）
- 三级紧急 → 全局告警 + 该队员在所有终端置顶 + 建议启动 Mayday 程序

#### Mayday程序触发后系统动作

```
FF触发Mayday（腕屏长按2秒）：
1. 立即通过所有可用链路广播位置（包括窄带）
2. 所有终端置顶红色告警「FF-03 MAYDAY」
3. GIS标记FF-03最后已知位置（永久保留直到手动解除）
4. 记录触发前30秒视频片段到日志
5. IC界面弹出「快速救援任务」预填单（仅需确认派发对象）
6. 自动通知后方医疗支援就位
```

### 6.4 搜救热力图规格

#### 热力图数据来源

| 来源 | 权重 | 说明 |
|-----|-----|-----|
| 队员GPS轨迹 | 基础层 | 覆盖半径按场景配置（室内3m/室外30m/热成像50m）|
| AI识别覆盖 | 叠加层 | 视频帧覆盖区域 |
| 毫米波扫描范围 | 叠加层 | 毫米波扇形扫描区 |
| 手动标记 | 强制层 | 指挥员手动标记「已完整搜索」|
| 疑似目标点 | 标注层 | AI告警点 |

#### 热力图状态色彩规范

- ⚪ 白色/浅灰：未搜索区域
- 🟡 黄色：已有队员经过但未系统性搜索
- 🟢 绿色：已完整搜索（达到覆盖密度阈值）
- 🔴 红色：发现目标/告警点
- 🟣 紫色：AI建议优先搜索区

## 十二、补充规格：任务前准备流程

> **目标**：从"设备开机"到"系统就绪可作战"，全程不超过 5 分钟。

### 12.1 标准化启动流程（SOP）

```
Step 1：设备开机与组网（~1min）
  - 背负台/车载台开机 → 自动广播发现邻节点 → Mesh 自组网建立
  - 系统首页显示：网络拓扑图 + 节点列表 + 信号强度
  - CT 确认所有关键节点在线（绿色）

Step 2：角色绑定与设备注册（~1min）
  - 各队员开机 → 扫码/NFC/账号登录 → 绑定设备（手持台/腕屏/腕表）
  - 系统显示：「FF-01 张伟 已绑定 → 手持台XX + 腕屏YY + 腕表ZZ」
  - IC 在笔记本端确认全队在线人员列表

Step 3：任务创建与地图加载（~1.5min）
  - IC 创建任务：输入任务名称 + 选择场景模板 + 上传/选择建筑平面图
  - GIS 地图自动加载楼层平面图，初始化搜救网格（全部标记为「未搜」）
  - 选择告警配置模板（或自定义）

Step 4：区域分配与任务下发（~1min）
  - IC 在地图上拖拽分配各班组负责区域
  - 一键下发给各TL → TL PAD收到 → TL进一步拆分到队员
  - 各FF腕屏震动并显示「负责区域：XX」

Step 5：就绪确认（~30s）
  - 系统显示：进场就绪检查单
    ✅ 网络覆盖验证
    ✅ 全员位置可见
    ✅ 视频回传正常（至少N路）
    ✅ 任务全员已回执
  - IC 点击「开始作战」，系统开始计时并记录完整数据
```

### 12.2 快速启动模式（紧急情况）

当无法完整执行上述流程时，支持「快速启动」：
- 省略角色绑定（以设备ID代替姓名，事后补录）
- 省略GIS地图加载（使用卫星图 + 手动网格划分）
- 省略任务模板选择（使用上次相同场景的配置）
- **不可省略**：组网确认、SOS链路测试、位置可见性验证

## 十六、补充规格：离线与降级模式

### 16.1 三级降级模式定义

| 模式 | 触发条件 | 可用能力 | 不可用能力 |
|-----|---------|---------|----------|
| **全功能模式** | 主链路正常 | 全部功能 | — |
| **保底模式** | 主链路中断，窄带可用 | 位置+心率+SOS+事件卡（文字）| 视频/语音流/高频识别 |
| **完全离线模式** | 主链路+窄带均中断 | 本地缓存任务/本地GIS/本地SOS记录 | 所有实时传输 |

### 16.2 完全离线时各终端行为

**笔记本端（IC）**：
- 显示最后已知队员位置（带时间戳）
- 任务可继续查看/编辑，断网恢复后自动同步
- 自动记录中断时段，复盘时标注数据缺口

**PAD端（TL/FF）**：
- 仍可查看最后下发的任务
- 语音上报内容本地缓存，网络恢复后自动上传
- SOS本地记录，网络恢复后立即上传（带原始触发时间戳）

**腕屏端（FF）**：
- 任务信息本地存储，继续显示
- SOS触发：本地记录 + 蓝牙广播（附近设备接力上传）

### 16.3 数据恢复同步策略

```
网络恢复后自动执行：
1. 上传离线期间所有 SOS 记录（带原始时间戳，优先级最高）
2. 上传位置轨迹（补全热力图和轨迹记录）
3. 上传语音事件缓存（可能触发延迟的事件卡生成）
4. 同步任务状态变化（回执、完成等）
5. 通知 IC 数据缺口时段及数据完整性状态
```

| 搜救覆盖分析 | 热力图最终态/覆盖率/未搜区域清单/AI建议覆盖路线对比 |
| 目标发现统计 | AI识别命中数/误报数/人工确认数/最终处置结果 |
| 队员安全记录 | 各队员心率血氧曲线/异常告警记录/SOS触发情况 |
| 通信链路报告 | 主链路质量/保底切换次数和时长/数据缺口 |
| 任务闭环分析 | 任务下发数/按时回执率/超时任务清单/原因分析 |
| 改进建议 | AI自动生成：本次搜救效率对比历史数据的差异点 |

### 17.2 演练模式（仿真模式）

**仿真模式特点**：
- 不连接真实设备，使用仿真数据源（预录制视频/仿真位置数据）
- 所有功能与实战模式完全一致
- 适用场景：指挥员培训、新系统演示、场景推演

**仿真数据集要求**：
- 预置至少5套标准场景的仿真数据（与典型场景一一对应）
- 支持「慢放/快放/暂停/回退」时间轴控制
- 支持自定义事件注入（在指定时刻触发指定告警）

## 二十、人员-多设备绑定机制

### 20.1 核心设计：混合绑定模式

采用「**预配置档案 + 现场主设备锚定**」的混合方案：

```
平时（后台配置阶段）：
  管理员在「人员装备档案」中维护每人的默认设备清单
  每人有一个"主设备"（通常是手持台）+ 若干"附属设备"

任务当天（现场激活阶段）：
  FF-03 在手持台上扫码/账号登录
    ↓
  手持台作为主锚点，通过以下方式拉起其他设备：
    - 蓝牙 BLE：自动发现并握手腕屏、腕表
    - Mesh 网络广播：通知摄像头、耳机节点加入"FF-03设备簇"
    ↓
  各附属设备收到绑定命令后，上报数据时自动携带 FF-03 的身份标识
    ↓
  指挥台看到的所有 FF-03 数据（位置/心率/视频/任务），均来自该设备簇的聚合
```

### 20.2 设备簇（Device Cluster）数据模型

```json
{
  "clusterId": "CLU-FF03-20240413",
  "personId": "FF-03",
  "personName": "张伟",
  "teamId": "TL-01",
  "activatedAt": "2024-04-13T08:05:00Z",
  "primaryDevice": {
    "deviceId": "HH-07",
    "type": "handheld",
    "status": "online",
    "battery": 78,
    "signal": -65
  },
  "attachedDevices": [
    { "deviceId": "WS-12", "type": "wristscreen", "bindingMethod": "ble", "status": "online", "battery": 92 },
    { "deviceId": "WT-05", "type": "smartwatch",  "bindingMethod": "ble", "status": "online", "battery": 65 },
    { "deviceId": "CAM-09","type": "chest_cam",   "bindingMethod": "mesh","status": "online", "battery": 45 },
    { "deviceId": "EAR-02","type": "earpiece",    "bindingMethod": "wired","status": "online","battery": null }
  ],
  "dataStreams": {
    "location": "HH-07",       // 位置数据来源
    "vitals":   "WT-05",       // 生命体征来源
    "video":    "CAM-09",      // 视频来源
    "voice":    "EAR-02",      // 语音来源
    "commands": ["HH-07","WS-12"]  // 任务/指令推送目标（多端同时推）
  }
}
```

**关键设计原则**：
- **一人一簇**：每个人对应唯一的活跃设备簇，数据在平台层聚合为"一个人的信息"
- **主设备失联降级**：主设备离线时，系统自动将信号最强的附属设备提升为临时主设备
- **数据来源透明**：指挥台可查看任意数据的来源设备，便于排查异常

### 20.3 三种绑定触发方式

#### 方式一：主设备登录自动拉取（推荐日常使用）

```
FF-03 在手持台登录
  → 系统加载其档案中的附属设备列表
  → 发送 BLE 广播：「我是 FF-03，编号 HH-07，请附近的 WS-12 / WT-05 应答」
  → WS-12 / WT-05 收到广播，确认配对 → 绑定完成
  → CAM-09 通过 Mesh 网络广播告知指挥台「我属于 FF-03」
```

**适用**：设备固定，日常任务，有时间正常上线

#### 方式二：指挥台手动分配（适合临时调整）

```
IC/CT 在笔记本端「设备总览」页：
  → 拖拽 CAM-11（未绑定）到 FF-03 的设备列表
  → 点击「确认绑定」
  → 系统通过 Mesh 网络推送绑定命令给 CAM-11
  → CAM-11 收到命令，后续上报数据时携带 FF-03 标识
```

**适用**：现场设备临时调配，或摄像头无法自主登录

#### 方式三：扫码快速绑定（适合紧急or替换场景）

```
新设备或替换场景：
  CT 扫设备二维码 → 在「设备准入」页选择「绑定人员」→ 选 FF-03
  → 输入替换原因 → 确认 → 原设备解绑 + 新设备绑定
  （整个流程 < 30秒）
```

**适用**：设备损坏紧急替换，设备临时借用

### 20.4 装备包（Kit）配置规格

**装备包**是一组设备的预设组合，按角色和场景预定义，降低现场配置成本：

| 装备包名称 | 包含设备类型 | 适用角色 | 适用场景 |
|----------|-----------|---------|---------|
| 城市内攻标配 | 手持台+腕屏+腕表+胸挂摄像+耳机 | FF | 建筑内攻 |
| 班组长增强包 | 手持台+PAD+腕屏+热成像+腕表 | TL | 所有场景 |
| 山地搜救包 | 背负台+腕屏+腕表+胸挂摄像+耳机 | FF | 室外搜救 |
| 危化侦察包 | 手持台+腕屏+腕表+胸挂摄像（防爆）| FF | 危化现场 |
| 轻装快速包 | 手持台+腕表 | FF | 紧急响应 |

**装备包使用流程**：
1. CT 提前在「装备包配置」页创建/编辑装备包
2. 任务创建时，IC 选择「本次任务使用装备包：城市内攻标配」
3. 系统为每个参战人员自动套用该装备包的要求
4. 现场设备绑定时，系统按装备包校验：缺少关键设备时警告提示

### 20.6 设备绑定异常处理

| 异常情况 | 系统响应 | 通知对象 |
|--------|---------|---------|
| 腕表持续5分钟无心率数据 | 告警「FF-03 腕表数据中断」，建议检查佩戴 | TL + IC |
| 主设备（手持台）离线 | 自动尝试通过腕屏/摄像头维持数量基本状态上报 | CT + IC |
| 同一设备同时被两人声明绑定 | 拒绝第二次绑定，提示「该设备已绑定XX」| 操作者 |
| 设备绑定后10分钟无数据上报 | 提示「设备可能未开机或信号覆盖不足」| CT |
| 未绑定设备上报数据 | 数据标记为「未识别设备」，提示CT处理 | CT |
