## Q01 · 参数化配置与方案细化

**日期**：2026-04-13  
**提问**：你提出的几个问题是否可以参数化可进行配置？另外我提供的系统方案有补充细化的地方吗？

**回答要点**：
1. **参数化配置可以且必须设计**，分三层：
   - **全局配置**（系统管理员）：AI识别阈值、告警阈值、链路切换条件等18项
   - **任务级配置**（指挥员）：本次AI识别类型、震动强度、超时时间、语音事件化开关等
   - **角色个人配置**：激活词、界面密度、告警偏好等
2. **方案可补充的缺口**：
   - 任务前准备流程（SOP）
   - 权限层级管理（RBAC）
   - 外部系统对接规范（119/应急平台）
   - 设备全生命周期管理
   - 离线降级模式与数据恢复
   - 复盘与战训系统
   - 多任务并发管理

**文档更新**：产品规格细化文档新增第十一~十八章（参数化配置体系、SOP流程、权限管理、外部对接、设备管理、离线降级、复盘战训、多任务并发）

## Q03 · 设备接入页面与多设备绑定机制

**日期**：2026-04-13  
**提问**：设备是接入的页面有涉及吗？而且比如一个消防人员佩戴了多个不同设备，这个绑定关系是怎么实现或者配置的呢？

**回答要点**：

**问题一：设备接入页面**
- 当前UI草图**未涉及**设备管理/接入页面，是真实缺口
- 补充了4个子页面规格：设备总览、人员装备档案、装备包配置、设备准入审核
- 设备总览：按在线状态/电量/信号排列的全设备表格视图，低电量<30%自动置顶橙色

**问题二：多设备绑定（以FF-03为例：手持台+腕屏+腕表+胸挂摄像+耳机）**

推荐**混合绑定模式**（预配置档案 + 现场主设备锚定）：
```
手持台 = 主锚点
  → 账号登录
  → BLE广播拉起腕屏/腕表
  → Mesh广播通知摄像头加入"设备簇"
  → 数据在平台聚合为"一个人的信息"
```

三种触发方式：
| 方式 | 场景 | 耗时 |
|-----|-----|-----|
| 主设备登录自动拉取 | 日常出发 | ~30s |
| 指挥台手动拖拽分配 | 临时调配 | 即时 |
| 扫码快速替换 | 设备损坏 | <30s |

**设备簇（Device Cluster）**中明确每个数据流来源：位置来自手持台、心率来自腕表、视频来自摄像头，任务同时推送到手持台和腕屏。

主设备失联时，自动将信号最强的附属设备提升为临时主设备。

**文档更新**：产品规格文档新增第十九章（设备接入管理页面）、第二十章（人员-多设备绑定机制）

## Q04 · 语音转写/指令/上报的设计与交互场景

**日期**：2026-04-13  
**提问**：语音转写 这块儿的设计有提供吗？语音指令、语音上报 这块儿的设计以及交互场景有提供吗？

**回答要点**：

原有文档仅有语音事件结构化字段和性能指标，缺少以下设计，现已补充：

**6.2-A 语音转写（ASR Transcription）**
- 数据流：麦克风→降噪→本地ASR→平台分流（字幕/关键词/存档）
- PC端转写栏UI：关键词橙色高亮`[地点]`；按角色颜色区分（FF冷灰/TL淡蓝/IC橙色）
- PAD端：单行精简，点击展开半屏历史
- 腕屏端：不显示转写，仅接收结构化推送
- 支持全文检索（关键词/时段），联动视频跳转，导出TXT/SRT/JSON

**6.2-B 语音指令（Voice Command）**
- 设计原则：固定意图短句，不做自由问答（高噪环境下最优）
- 激活：说「指令」/长按PTT/腕屏侧键/PC快捷键 `Alt+V`
- 指令集18条，分4类：上报类/位置类/查询类/操作类
- 状态机：待机→监听(5s)→确认(置信度<75%要求重复)→执行→语音反馈
- **语音反馈必须有**（用户可能看不到屏幕）

**6.2-C 语音上报（Voice Reporting）**
- 4种上报模式：结构化（主流）/快捷/自由/被动呼救检测
- 结构化上报7步流程：激活→录音(5s)→ASR转写→NLP解析→确认播报→队员确认→IC处理
- 自由上报生成"低置信度待确认"事件卡，含原始语音回放链接
- 呼救词词库分高/中优先级，触发需IC人工确认才生成事件卡（防误报）

**6.2-D 语音频道管理**
- 4类频道：指挥/班组/全员广播/点对点
- PTT/HOT MIC/静默三档
- IC强制查听功能（查岗，会告知被监听队员）

**文档更新**：产品规格文档 §6.2 新增 6.2-A~D 四个子章节（约350行）

## Q05 · Mesh 板卡能否支持语音转写

**日期**：2026-04-13  
**提问**：mesh板卡能支持语音转写？

**回答要点**：

**不能，也不应该。**

**原因**：
- Mesh 板卡是**射频通信专用芯片**，资源全分配给协议处理（TDMA/路由/加解密）
- CPU ~400MHz~1GHz（RF专用），RAM 64~256MB
- Whisper-tiny（最轻量ASR）需 1~2 GFLOPS 持续算力，差距约10~20倍
- 强行运行ASR会与Mesh协议争抢资源，导致通信丢包、组网不稳定

**正确架构**：**双芯片/双板架构**
```
设备内部：
  [应用处理器 AP] → 音频采集 → 本地ASR → 文字
                                              ↓
  [Mesh 板卡]    → 传输文字(几十字节)  ← 而非传音频(32~64KB/s)
```

**带宽收益**：ASR在AP端处理，Mesh只传文字，节省带宽约99%

**各设备分级**：
| 设备 | ASR能力 |
|-----|---------|
| 手持台（有AP）| ✅ Whisper-tiny 级本地ASR |
| 背负台 | ✅ Whisper-base 级，可作区域代理节点 |
| 车载台/笔记本 | ✅ 完整模型，全区兜底代理 |
| 腕屏/腕表 | ❌ 只收文字推送 |
| **Mesh 板卡** | **❌ 禁止，破坏通信稳定性** |

**弱AP设备的代理机制**：
音频→Mesh传输压缩音频→背负台/车载台代理推理→文字回传来源设备

**文档更新**：产品规格文档 §6.2 性能要求部分增加硬件架构说明、设备ASR能力分级表、弱AP代理机制

## Q07 · 地图引擎选型（离线/无跨网络要求）

**日期**：2026-04-13  
**提问**：地图使用什么方案？我有看到 Leaflet + React Leaflet + Esri 卫星底图这个方案，你还有更好的推荐吗？最好没有跨网络的要求

**关键约束**：系统运行在 Mesh 自组网局域网环境，**无互联网连接**，地图方案必须完全离线可用。

**回答要点**：

**Leaflet + Esri 为什么不行**：
- Esri 卫星瓦片依赖外部 CDN（`server.arcgisonline.com`），必须联网
- 在 Mesh 自组网局域网中，该请求出不去，地图显示为空白
- 即使 Leaflet 本身开源，也因底图依赖而无法离线

**推荐方案：MapLibre GL JS + 本地 TileServer GL**

理由：
- **100% 开源免费**：MapLibre GL JS 是 Mapbox GL JS 的开源分叉（Mapbox 2020年改商业授权后分叉），无 API Key，无联网要求
- **WebGL 渲染**：GPU 加速，性能远优于 Leaflet（DOM渲染）
- **完全离线**：卫星底图打包为 `.mbtiles` 文件，通过本地 TileServer GL 服务提供
- **带宽低**：矢量瓦片远小于栅格瓦片
- **功能完整**：热力图（原生支持）、GeoJSON叠加、实时标记更新无性能损耗

**性能对比**（关键）：
```
Leaflet (DOM渲染)：
  每个 Marker = 一个 <div>，100个队员 = 100个DOM，频繁更新有抖动

MapLibre GL JS (WebGL渲染)：
  所有元素在 GPU 渲染，1000个标记无感知，每帧更新流畅
→ 实时指挥系统选 MapLibre
```

**卫星底图离线制作方式**：
| 方式 | 工具 | 说明 |
|-----|-----|-----|
| 按区域下载 | QGIS（免费）| 国内主流，导出 .mbtiles |
| 天地图预缓存 | 自定义爬虫 | 国内卫星图，注意版权 |
| 购买商业离线图 | MapTiler Engine | 一键生成高质量包 |
| 自有遥感图像 | GDAL 转换 | 最高分辨率 |

**推荐技术栈（更新版）**：
```
maplibre-gl        # 地图引擎（替代 Mapbox GL）
react-map-gl       # React 封装（v7+ 支持 MapLibre 后端）
tileserver-gl      # Docker 本地瓦片服务（读取 .mbtiles）
```
